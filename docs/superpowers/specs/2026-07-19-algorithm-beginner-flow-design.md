# アルゴリズム初心者向け7ステップ再設計

## 目的

`tech-algorithm-flowchart` を、IT未経験者が「アルゴリズム＝目的を達成するための手順」と日常感覚から理解できる学習体験へ再設計する。変数・代入・繰り返し・条件分岐・フローチャートを同時に提示せず、日常の手順から試験表現まで一段ずつ橋渡しする。

## 対象範囲

- `tech-algorithm-flowchart` 専用体験を、7ステップのクライアントコンポーネントへ置き換える。
- 既存のトピックID、レジストリ、確認問題、完了判定、XP、進捗保存、復習連携は維持する。
- STEP7内に3問のミニ確認問題を追加する。これは理解の仕上げ用であり、既存の4問確認問題による採点・レッスン完了・進捗保存を置き換えない。
- 他トピックの専用体験および共通 `ExperienceSlideDeck` の挙動は変更しない。

## 採用方式

対象トピック内に専用の7ステップUIを実装する。親の `AlgorithmExperience` が現在位置と各体験の状態を保持し、現在ステップに対応する子コンポーネントだけを条件描画する。

共通 `ExperienceSlideDeck` は多数のトピックで使用され、高さ安定化のため非表示パネルもマウントしている。今回これを変更すると他トピックへ波及するため、アルゴリズム専用体験は複数の共通 `Panel` を登録せず、内部ナビゲーションで完結させる。

## 学習フロー

### STEP1：手順を並べる

「カップ麺を作る順番を並べよう」という課題を表示する。モバイルとキーボード操作を考慮し、ドラッグ操作ではなく候補を順番にタップして回答欄へ追加する。

正しい順番は次のとおり。

1. ふたを開ける
2. お湯を入れる
3. 3分待つ

3つを選択後に正誤を表示する。間違えた場合はやり直せる。正解後に「目的を達成するための手順をアルゴリズムと呼びます」と初めて用語を提示する。正解するまで次のステップへ進めない。

### STEP2：順番が重要

「3分待つ → お湯を入れる → ふたを開ける」という誤った順番を表示し、「うまくいく」「うまくいかない」の二択で答える。回答後に「手順は内容だけでなく順番も重要」と説明する。回答するまで次へ進めない。

### STEP3：コンピュータの処理

次の4処理を、最初は未実行の一覧として表示する。

1. 3を受け取る
2. 2を受け取る
3. 足す
4. 5を表示

「1行実行する」を押すごとに次の1行だけが実行済みになり、現在行を明示する。全行実行後に「コンピュータも手順を上から順番に実行する」と説明する。この段階ではフローチャート、変数、代入記号を表示しない。全行実行するまで次へ進めない。

### STEP4：数字を入れる箱

「変数」という用語より先に、数字を保存できる「名前の付いた箱」として説明する。「合計の箱」を0から開始し、「1を足す」「2を足す」の操作で0 → 1 → 3へ変化させる。

体験後に次の対応を表示する。

- 名前の付いた箱を「変数」と呼ぶ。
- 「合計の箱を0にする」は `合計 ← 0` と書く。
- `←` は右側の値を左側の箱へ入れる印であり、等号ではない。

### STEP5：繰り返し

画面上の主要な数値表示を「合計」と「現在の数字」の2つに限定する。初期値は合計0、現在の数字1とする。

「現在の数字を足す」を押すごとに、合計へ現在の数字を加え、現在の数字を1増やす。1から5までを順番に加えると合計15になり、「同じ処理を何度も行うことを繰り返しと呼ぶ」と説明する。5回の操作を終えるまで次へ進めない。

### STEP6：フローチャート

ここで初めてフローチャートを表示する。処理は平易な日本語で次の順に並ぶ。

1. 開始
2. 合計を0にする
3. 数字を足す
4. 5以下か確認
5. 次の数字へ
6. 合計を表示
7. 終了

通常画面は現在位置を中心に「前・現在・次」の最大3処理だけを表示し、「処理を進める」「処理を戻す」で追跡できる。先頭と末尾では存在する処理だけを表示する。

「全体図を見る」で全フローチャートをモーダル表示する。モーダルは `role="dialog"`、`aria-modal="true"`、明示的な見出しと閉じるボタンを持ち、背景クリックと Escape キーでも閉じられる。全体図の各処理は縦に並べ、現在位置を強調する。

### STEP7：試験で使う正式な表現

次の対応を並べて表示する。

- 合計の箱を0にする ⇔ `合計 ← 0`
- 合計に現在の数字を足す ⇔ `合計 ← 合計 + i`
- 次の数字へ ⇔ `i ← i + 1`

その後、選択式のミニ確認問題を3問、1問ずつ表示する。

1. 「合計 ← 0」が表す意味
2. `合計 ← 合計 + i` が表す意味
3. `i ← i + 1` が表す意味

各問は回答直後に正誤と短い説明を表示し、回答後に次の問題へ進める。3問すべて回答すると、日常の手順から試験表現までつながったことを伝える。誤答しても学習フローを止めず、意味を確認して先へ進める。

## UI構造

### ステップシェル

常に次の要素だけを描画する。

- `STEP n / 7`
- 現在位置を示す7分割の進捗バー
- 現在ステップの短いタイトル
- 現在ステップの内容
- 「戻る」「次へ」

「戻る」はSTEP1で無効、「次へ」はSTEP7で完了表示にする。体験が必須のステップでは、必要操作が終わるまで「次へ」を無効化し、理由をボタン付近に短く示す。

### 描画と高さ

ステップ定義の全要素をDOMへ並べてCSSで隠す構造は使わない。現在のステップ番号で分岐し、該当する1コンポーネントだけをReactツリーへ描画する。したがって非表示ステップはDOMにもレイアウト計算にも存在しない。

ステップの状態は親で保持するため、戻った場合も並べた順番、計算結果、回答内容を維持する。ページ再読み込み後の内部ステップ再開は新設しない。既存の進捗保存はレッスン確認問題の完了時に従来どおり行う。

### モバイルとレスポンシブ

- 390px前後の画面幅を基準に、本文、主要操作、ナビゲーションが一度に把握できる密度にする。
- 固定ピクセル幅を避け、カードは `w-full` と狭い余白を使う。
- 操作ボタンはタップしやすい高さを確保し、横幅が狭い場合も折り返しや縦配置で欠けないようにする。
- フローチャート全体図だけは内容上縦スクロールを許容するが、通常ステップには全体図を置かない。

## 状態とデータフロー

親コンポーネントが次の状態を保持する。

- 現在の学習ステップ（1〜7）
- STEP1の選択順と正解済みフラグ
- STEP2の回答
- STEP3の実行済み行数
- STEP4の箱の合計と実行済み操作数
- STEP5の合計と現在の数字
- STEP6の現在処理位置とモーダル表示状態
- STEP7の現在問題と回答結果

各ステップ子コンポーネントは必要な値とイベントだけをpropsで受け取る。学習内容の定数、フローチャート処理一覧、正式表現の対応、ミニ確認問題はファイル上部の読み取り専用データとして定義する。

## アクセシビリティ

- すべての操作を `button` で実装し、色だけで状態を伝えない。
- ステップ切り替え時の見出しを一意にし、進捗にはテキスト表現も付ける。
- 実行結果・正誤フィードバックには `aria-live="polite"` を使用する。
- 無効な「次へ」には視覚的な無効状態と補足文を付ける。
- フローチャートの線や装飾は `aria-hidden` とし、処理内容はテキストで読み上げられるようにする。

## テスト戦略

VitestとReact Testing Libraryで対象コンポーネントの利用者視点のテストを追加する。

- 初期表示はSTEP1のみで、STEP6のフローチャートやSTEP7の正式表現がDOMに存在しない。
- カップ麺の手順を正しく並べるとアルゴリズムの定義が表示され、次へ進める。
- STEP2の回答後に順番の重要性が表示される。
- STEP3は1クリックにつき1行だけ進み、4行後に説明が表示される。
- STEP4は箱が0 → 1 → 3と変化し、その後に変数と代入表現が表示される。
- STEP5は5回の操作で合計15、現在の数字6となり、繰り返しの説明が表示される。
- STEP6の通常表示は最大3処理で、全体図はモーダルを開いたときだけDOMに現れる。
- STEP7は平易な表現と正式表現を対応付け、3問すべてに回答できる。
- 「戻る」で前ステップに戻っても親が保持する操作状態は失われない。

既存のトピックデータおよび完了クイズのテストも実行し、`tech-algorithm-flowchart` のID、4問の既存確認問題、進捗保存経路が変わっていないことを確認する。

## 完了条件

- 日常例から始まり、フローチャートはSTEP6まで表示されない。
- 変数を「名前の付いた箱」として体験後に正式名称へ接続する。
- 1〜5の合計操作で繰り返しを体験できる。
- 平易な処理からフローチャート、試験表現へ自然につながる。
- 初期表示が縦長にならず、非表示ステップがDOMと高さ計算に含まれない。
- 既存デザイン、レスポンシブ、確認問題、進捗保存を維持する。
- 対象外トピックのコードと挙動を変更しない。
- TypeScript、lint、対象テスト、全テスト、production buildが成功する。
